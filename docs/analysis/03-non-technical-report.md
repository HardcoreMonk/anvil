## Ephemera가 하는 일

Ephemera는 AI 작업자를 매번 새로 만든 작은 가상 컴퓨터 안에 넣어 실행하는 시스템이다. 작업이 끝나면 그 가상 컴퓨터를 없애고, 디스크와 네트워크 흔적도 정리한다.

쉽게 말하면 다음과 같다.

- AI 작업을 그냥 내 서버에서 직접 실행하지 않는다.
- 작업마다 별도의 작은 방을 만든다.
- 그 방 안에 AI 도구를 넣는다.
- 작업이 끝나면 방을 통째로 치운다.

이렇게 하면 한 작업이 다른 작업이나 host 서버에 영향을 줄 가능성을 줄일 수 있다.

## 등장인물

### Control Plane

전체 관리자다. 실제 파일은 `cmd/goose-daemon`이다.

하는 일:
- 새 가상 컴퓨터 만들기
- 현재 실행 중인 가상 컴퓨터 목록 보여주기
- 가상 컴퓨터 삭제하기
- 필요한 디스크, 네트워크, 실행 파일 준비하기

### MicroVM

아주 작은 가상 컴퓨터다. Ephemera는 Firecracker라는 기술을 사용한다. Firecracker는 AWS Lambda 같은 서버리스 환경에서도 쓰이는 경량 VM 기술이다.

MicroVM 안에는 Debian이라는 Linux 운영체제가 최소 구성으로 들어간다.

### goose-agent

MicroVM 안에서 실행되는 작은 HTTP 서버다. 실제 파일은 `cmd/goose-agent`다.

하는 일:
- "이 prompt로 AI 작업을 실행해줘"라는 요청을 받는다.
- Goose라는 AI agent 도구를 실행한다.
- 결과를 JSON으로 돌려준다.
- 현재 바쁜지 확인해준다.
- 종료 요청을 받으면 스스로 멈춘다.

### Golden Image

가상 컴퓨터의 원본 디스크다.

복사기 원본이라고 생각하면 된다. 새 VM을 만들 때마다 원본을 직접 쓰지 않고, 원본 디스크를 복사해서 각 VM 전용 디스크를 만든다. 그래서 VM끼리 파일 상태가 섞이지 않는다.

### Goose

VM 안에서 실제 AI 작업을 실행하는 도구다. Ephemera는 Goose를 직접 구현하지 않고, VM 안에 Goose 실행 파일을 넣어 사용한다.

## 큰 흐름

### 1. 시스템 시작

관리 프로그램인 `goose-daemon`을 실행한다.

처음 실행하면 시간이 걸릴 수 있다.

- VM 안에 들어갈 `goose-agent`를 빌드한다.
- VM 원본 디스크를 만든다.
- Firecracker 실행 파일을 내려받는다.
- Linux kernel 파일을 내려받는다.
- 네트워크 bridge를 만든다.

두 번째 실행부터는 이미 만든 파일이 있으면 재사용한다.

### 2. VM 만들기

사용자가 `POST /vms`라는 API를 호출하면 새 VM을 만든다.

내부에서는 다음 일이 일어난다.

1. VM 이름을 정한다.
2. VM에 쓸 내부 IP를 배정한다.
3. VM 전용 네트워크 장치를 만든다.
4. golden image를 복사해 VM 전용 디스크를 만든다.
5. AI provider 설정과 API key를 VM 디스크 안에 넣는다.
6. Firecracker로 VM을 켠다.
7. VM 안의 `goose-agent`가 응답할 때까지 기다린다.
8. 사용자에게 VM 주소를 알려준다.

### 3. AI 작업 실행

사용자는 VM 생성 응답에서 받은 `agent_url`로 작업을 보낸다.

예:

```json
{"prompt":"현재 시스템 환경을 확인해줘"}
```

그러면 VM 안의 `goose-agent`가 Goose를 실행하고 결과를 돌려준다.

한 VM에서는 한 번에 하나의 작업만 실행한다. 이미 작업 중이면 "바쁘다"는 응답을 준다.

### 4. VM 삭제

사용자가 `DELETE /vms/{vm_id}`를 호출하면 다음을 정리한다.

- Firecracker VM process
- VM 전용 디스크
- VM 네트워크 장치
- VM IP 주소
- 임시 socket/log 파일

즉, 작업 공간을 통째로 버리는 방식이다.

## 왜 이렇게 복잡하게 만들까?

AI agent는 파일을 읽고, 명령을 실행하고, 인터넷에 접근할 수 있다. 강력하지만 위험할 수도 있다.

그래서 Ephemera는 AI agent를 host 서버에 직접 풀어놓지 않고, 매번 격리된 작은 VM 안에서 실행한다.

장점:
- 작업끼리 서로 영향을 덜 준다.
- 작업이 끝나면 디스크를 삭제할 수 있다.
- VM 경계가 컨테이너보다 강한 격리층을 제공한다.
- 설정과 API key를 VM 생성 시점에 주입하므로 image를 매번 다시 만들 필요가 없다.

## 웹 화면은 있는가?

없다.

이 프로젝트에는 사용자용 웹 프론트엔드가 없다. 버튼을 누르는 화면 대신 HTTP API를 제공한다. 사람이 직접 쓰려면 `curl` 같은 명령어를 쓰거나, 별도의 웹 UI/client를 만들어 붙여야 한다.

## 보안 관점에서 중요한 점

좋은 설계:
- 기본적으로 관리 API는 내 컴퓨터 안에서만 접속되는 `127.0.0.1:3000`에 열린다.
- API token을 설정할 수 있다.
- VM마다 디스크를 복사해서 따로 쓴다.
- VM이 삭제되면 디스크와 네트워크도 정리한다.
- 실제 API key 파일은 git에 올라가지 않도록 되어 있다.

주의할 점:
- API token을 설정하지 않으면 관리 API가 인증 없이 열린다.
- VM 안의 `goose-agent`에는 별도 인증이 없다.
- 외부에 공개하려면 HTTPS reverse proxy와 token 설정이 필요하다.
- AI provider API key가 VM 디스크 안에 들어가므로 VM 접근 권한 관리가 중요하다.

## 운영할 때 필요한 환경

일반 노트북 앱처럼 아무 데서나 실행되는 프로그램은 아니다.

필요한 것:
- Linux 서버
- KVM 사용 가능 환경
- root 또는 sudo 권한
- Firecracker 실행 가능 커널/CPU
- 디스크 이미지 생성 도구
- Linux 네트워크 설정 도구

이 때문에 서버 운영 지식이 필요하다.

## 이 프로젝트의 현재 성숙도

작동하는 첫 버전 형태에 가깝다.

이미 갖춘 것:
- VM 생성/삭제 API
- VM 내부 agent
- golden image 자동 생성
- Firecracker 자동 다운로드
- 네트워크/IP/TAP 관리
- 간단한 end-to-end 테스트 스크립트
- README 문서

더 보강하면 좋은 것:
- 자동화된 Go 단위 테스트
- 더 강한 다운로드 파일 검증
- token 변경 시 재시작 없이 반영
- VM 생성 ID 충돌 방지
- VM 내부 agent 인증 강화
- 운영 모니터링/로그 구조화
- 웹 UI 또는 공식 client

## 비즈니스 관점 해석

Ephemera는 "AI agent를 안전한 일회용 작업실에서 실행하는 인프라"다.

활용 가능성이 있는 분야:
- 코드 실행형 AI agent
- 고객별 격리 작업 공간
- 민감한 자동화 작업
- 실험 후 흔적을 남기지 않는 분석 환경
- agent sandboxing 플랫폼

다만 현재 코드는 단일 host 기반이다. 여러 서버에 VM을 나눠 띄우는 클러스터 기능은 없다. 대규모 서비스로 만들려면 scheduling, quota, monitoring, billing, audit log, multi-host orchestration 같은 기능이 추가되어야 한다.

## 최종 요약

Ephemera는 사용자가 요청할 때마다 작은 가상 컴퓨터를 만들고, 그 안에서 AI agent 작업을 실행한 뒤, 작업 공간을 삭제하는 백엔드 시스템이다. 화면이 있는 제품이라기보다, 다른 서비스가 붙어 사용할 수 있는 인프라 API에 가깝다. 핵심 가치는 "격리", "일회성", "자동 준비", "자동 정리"다.
