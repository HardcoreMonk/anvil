# ephemera 0.2.0 변경 분석: 0.1.0 대비

## 기준

- 분석 대상: `ephemera` runtime
- anvil 관점: ephemera 0.2.0은 IronClaw 결합 프로젝트의 기반 runtime이다.
- 공식 저장소: `https://github.com/HardcoreMonk/anvil/`
- 0.1.0 기준 커밋: `157753f`
- 0.2.0 릴리즈 커밋: `abcaa86`
- 비교 범위: `157753f..abcaa86`

원격 저장소에는 분석 시점 기준으로 `0.1.0`, `0.2.0` Git tag가 없었다. 따라서 `0.2.0`은 `Merge pull request #4` 커밋을 릴리즈 기준으로 보았고, 바로 이전 메인라인 머지 커밋 `157753f`를 0.1.0 기준점으로 사용했다.

## 변경 규모

0.2.0은 20개 파일에 걸쳐 약 3,603줄 추가와 376줄 삭제가 발생했다. 변경의 성격은 다음과 같다.

- 신규: CI, micro-init, snapshot storage, e2e 테스트, 단위 테스트
- 확장: daemon API, VM 복원 로직, guest agent 인증과 vsock 제어
- 변경: rootfs 빌드 스크립트, VM kernel args, README와 release note
- 삭제: 기존 단순 시나리오 테스트 스크립트

## 핵심 변화

### 1. Single-host feature complete 수준으로 범위 확대

0.1.0은 Firecracker VM을 만들고, guest agent에 직접 task를 보내며, 기본적인 stop/delete 흐름을 검증하는 PoC에 가까웠다.

0.2.0은 single-host 환경에서 운영에 필요한 주요 기능을 대거 추가했다.

- daemon을 통한 agent proxy
- per-VM agent token
- VM snapshot 생성
- snapshot restore
- diff snapshot
- COW rootfs restore
- guest IP 재설정
- API client hot reload
- CI와 테스트 체계

즉, 0.2.0의 변화는 "VM 실행 가능"에서 "VM 실행 상태를 저장, 복원, 재노출, 검증할 수 있음"으로 요약된다.

### 2. guest PID 1 도입

신규 파일 `cmd/micro-init/main.go`가 추가되었다.

기존에는 rootfs 안에서 agent 실행과 종료가 비교적 단순하게 처리되었다. 0.2.0에서는 guest 내부의 PID 1 역할을 하는 `micro-init`이 추가되어 다음을 담당한다.

- `/proc`, `/sys`, `/dev`, `/dev/pts` mount
- 기본 환경 변수 설정
- `/usr/local/bin/goose-agent` 실행
- SIGTERM, SIGINT 수신 시 agent 종료 전파
- `sync` 후 guest poweroff

이에 맞춰 `internal/vm/machine.go`의 kernel args도 `init=/usr/local/sbin/micro-init`을 사용하도록 변경되었다. `scripts/build_image.sh`는 빌드 산출물 `artifacts/micro-init`을 rootfs의 `/usr/local/sbin/micro-init`에 설치한다.

이 변경은 VM 삭제나 stop 요청 시 host와 guest의 종료 의미를 맞추는 데 중요하다.

### 3. VM별 agent 인증 추가

0.1.0에서는 daemon API 인증과 guest agent 접근 사이의 경계가 약했다. 0.2.0에서는 VM마다 agent token을 발급하고 rootfs 내부에 주입한다.

주요 변경점은 다음과 같다.

- `cmd/goose-daemon/api.go`
  - `generateAgentToken()` 추가
  - VM 생성 결과에 `agent_token` 포함
  - running VM 상태에 `agentToken` 저장
- `internal/storage/provisioner.go`
  - `VMPrepareOptions` 도입
  - `/root/.ephemera-agent-token` 파일을 `0600` 권한으로 작성
- `cmd/goose-agent/main.go`
  - token 파일 로딩
  - `/tasks`, `/stop`에 인증 middleware 적용
  - `/health`는 인증 없이 유지

이 구조는 daemon이 control-plane API token으로 외부 요청을 받고, 내부적으로 per-VM agent token을 사용해 guest agent에 접근하도록 만든다.

### 4. daemon agent proxy 도입

0.1.0에서는 클라이언트가 VM agent URL을 받아 guest agent에 직접 접근하는 구조에 가까웠다. 0.2.0에서는 daemon이 agent endpoint를 proxy한다.

신규 또는 확장된 API는 다음과 같다.

- `POST /vms/{vm_id}/tasks`
- `GET /vms/{vm_id}/health`
- `POST /vms/{vm_id}/stop`

daemon은 VM별 agent token을 자동으로 붙여 guest agent에 요청을 전달한다. `EPHEMERA_PUBLIC_URL`이 설정되어 있으면 VM 생성 응답의 agent URL도 private guest IP가 아니라 control-plane URL 기반으로 구성된다.

이 변화는 IronClaw나 MCP adapter 같은 상위 계층이 guest IP와 token을 직접 다루지 않아도 되게 만든다.

### 5. per-VM profile 지원

0.2.0에서 VM 생성 요청에 `profile` 필드가 추가되었다. daemon은 다음 경로를 profile별 설정으로 인식한다.

- `configs/profiles/{profile}/goose.yaml`
- `configs/profiles/{profile}/goose-secrets.yaml`

profile 이름에는 `/`, `\`, `..`이 허용되지 않는다. profile이 비어 있으면 기존 기본 설정 경로를 사용한다.

이 기능은 같은 host에서 서로 다른 LLM provider, model, secret 조합을 가진 VM을 분리 실행하기 위한 기반이다.

### 6. API token hot reload

`cmd/goose-daemon/main.go`는 SIGHUP을 처리해 API client 설정을 다시 읽는다. 기존 VM은 유지된다.

구조적으로는 다음이 바뀌었다.

- `ControlPlane`에 `clientsMu`, `clients` 저장
- 인증 middleware가 고정 slice가 아니라 `getClients()` 함수를 통해 최신 client 목록을 조회
- `ReloadClients()`가 설정 파일을 다시 로드

운영 중 API key 교체를 위해 daemon 전체 재시작이 필요하지 않게 된 점이 핵심이다.

### 7. snapshot과 restore 추가

가장 큰 변화는 snapshot subsystem이다. 신규 파일 `internal/storage/snapshot.go`가 추가되었고, daemon API에도 snapshot endpoint가 추가되었다.

신규 API:

- `GET /snapshots`
- `POST /vms/{vm_id}/snapshot`
- `POST /snapshots/{snapshot_id}/restore`
- `DELETE /snapshots/{snapshot_id}`

지원하는 snapshot type:

- `full`
- `diff`
- `auto`

`auto`는 같은 source VM의 첫 snapshot이면 full, 이후 snapshot이면 diff로 해석된다. 명시적으로 diff snapshot을 요청하려면 같은 source VM에 기존 full snapshot이 있어야 한다.

snapshot metadata에는 다음 정보가 저장된다.

- snapshot ID
- source VM ID
- profile
- snapshot type
- base snapshot ID
- guest IP
- TAP device
- vsock path
- MAC address
- agent token
- disk path
- memory snapshot path
- disk copy path
- 생성 시각

0.2.0의 restore는 단순 disk copy가 아니라 VM 네트워크와 agent 접근 정보까지 재구성하는 복원 흐름이다.

### 8. diff snapshot과 dirty page tracking

`internal/vm/machine.go`는 Firecracker snapshot 관련 설정을 확장했다. VM 생성 시 dirty page tracking을 활성화하고, diff snapshot 생성 시 Firecracker의 diff snapshot type을 사용한다.

restore 시에는 `MergeMemoryDiff`가 sparse memory diff를 full snapshot의 memory 파일 위에 overlay한다. 이 함수는 `SEEK_DATA`, `SEEK_HOLE`을 사용해 sparse 파일의 실제 데이터 구간만 병합한다.

이는 snapshot 저장 비용과 restore 시간을 줄이기 위한 핵심 개선이다.

### 9. COW rootfs restore

0.2.0은 snapshot restore 시 rootfs 전체를 매번 복사하지 않기 위해 Linux device-mapper snapshot을 사용한다.

주요 구성:

- base disk를 read-only loop device로 연결
- sparse exception store `.cow` 파일 생성
- COW loop device 연결
- `dmsetup create`로 `/dev/mapper/cow-*` 생성
- 생성된 mapper device를 VM disk path로 bind mount

device-mapper snapshot 구성이 실패하면 legacy bind mount fallback을 사용한다.

이 기능은 여러 restore VM이 같은 base snapshot에서 파생될 수 있게 해주는 기반이다. 단, 같은 snapshot의 동시 restore는 아직 명시적으로 지원 대상이 아니다.

### 10. guest IP 재설정

snapshot으로 복원된 VM은 memory snapshot 안에 과거 IP 상태를 가지고 있을 수 있다. 0.2.0은 restore 후 vsock을 통해 guest에 IP 변경 명령을 보낸다.

agent 측 변경:

- vsock listener 추가
- `CHANGE_IP <cidr_ip> <gateway>` 명령 처리
- `ip addr flush`
- `ip addr add`
- `ip link set eth0 up`
- `ip route replace default`

host 측 변경:

- `internal/vm/machine.go`에 `ReconfigureGuestIP`
- `internal/network/manager.go`에 `AllocateForRestore`

restore가 기존 snapshot의 IP, TAP, MAC 정보를 그대로 쓰는 것이 아니라 현재 host의 네트워크 상태에 맞게 재할당할 수 있게 되었다.

### 11. 테스트와 CI 강화

0.2.0은 테스트 체계가 크게 보강되었다.

신규 CI:

- `.github/workflows/ci.yml`
- `go build ./...`
- `go vet ./...`
- `go test ./...`

신규 단위 테스트:

- daemon profile path 검증
- agent token 생성 검증
- config 로딩 검증
- agent auth middleware 검증
- provisioner token file 권한 검증

신규 e2e:

- `e2e_test.sh`
- 단일 VM 생성, task, stop, delete
- 병렬 VM task
- full snapshot lifecycle
- restore와 token 보존
- diff snapshot
- COW rootfs sparse allocation
- agent proxy endpoint
- `EPHEMERA_PUBLIC_URL`

기존 `simple_test_scenario.sh`는 삭제되었다.

## API 관점의 변화

### VM 생성 응답

0.2.0의 VM 생성 응답은 기존보다 민감 정보와 routing 정보를 더 많이 포함한다.

- `vm_id`
- `guest_ip`
- `agent_url`
- `profile`
- `agent_token`

`agent_token`은 직접 agent에 접근해야 하는 하위 호환 시나리오를 위해 반환되지만, 일반적인 상위 계층은 daemon proxy endpoint를 사용하는 것이 낫다.

### VM endpoint 확장

기존 VM delete/snapshot 중심에서 다음 endpoint들이 추가되었다.

- task 실행
- health 조회
- stop 요청
- snapshot 생성

상위 계층은 guest IP를 몰라도 `POST /vms/{vm_id}/tasks`만 호출하면 된다.

### Snapshot endpoint 신설

0.2.0은 snapshot을 일급 리소스로 다룬다.

- list
- create from VM
- restore to new VM
- delete

이는 anvil의 "작업 세션을 저장하고 되살리는 실행 환경"이라는 방향과 직접 연결된다.

## 운영 관점의 변화

### 좋아진 점

- VM 종료가 guest 내부에서 정리된다.
- agent 접근이 VM별 token으로 보호된다.
- 외부 클라이언트는 daemon proxy를 통해 VM task를 실행할 수 있다.
- snapshot/restore가 API로 노출된다.
- diff snapshot과 COW rootfs로 저장 비용을 줄이는 방향이 생겼다.
- CI와 테스트가 도입되어 회귀 확인 기준이 생겼다.
- API client secret 교체가 daemon 재시작 없이 가능하다.

### 여전히 남은 제약

- multi-host scheduling은 없다.
- tenant quota나 정책 엔진은 없다.
- VM egress 제한은 없다.
- snapshot metadata에 agent token이 저장된다. 파일 권한은 `0600`이지만 민감 정보 수명 관리가 별도로 필요하다.
- 같은 snapshot의 동시 restore는 지원하지 않는다.
- device-mapper fallback 경로에는 원자성 측면에서 추가 검토가 필요하다.
- snapshot API의 일부 JSON decode 실패가 명확한 400 응답으로 처리되지 않는다.

## 위험 분석

### restore fallback 원자성

`restoreSnapshot`은 device-mapper snapshot 구성이 실패하면 bind mount 기반 fallback으로 내려간다. 그런데 fallback 경로에서는 bind mount 설정과 `RestoreMachine` 호출 사이의 lock 범위가 device-mapper 경로보다 약하다.

0.1 문서에서 이미 rootfs bind mount와 VM restore의 원자성이 중요하다고 봤다면, 0.2.0에서도 이 부분은 주의해야 한다. 특히 같은 snapshot 또는 같은 disk path에 대한 병렬 restore가 들어오면 fallback 경로가 의도보다 약한 격리를 제공할 수 있다.

### token 저장 위치

agent token은 VM 생성 시 rootfs 안에 쓰이고 snapshot metadata에도 저장된다. metadata 파일 권한은 `0600`이지만, snapshot 보관소 자체가 장기 보존되는 경우 token rotation, snapshot export, backup 시 민감 정보 취급 기준이 필요하다.

### direct agent URL 하위 호환

`agent_url`이 `EPHEMERA_PUBLIC_URL`에 따라 daemon proxy URL이 되거나 private guest IP URL이 될 수 있다. 클라이언트는 이 차이를 가정하지 말고 가능하면 daemon proxy API를 기준으로 통합하는 편이 안정적이다.

### diff snapshot 의존성

diff snapshot은 base full snapshot에 의존한다. 0.2.0은 base가 있는 diff snapshot이 존재하면 full snapshot 삭제를 막는다. 이 정책은 맞지만, 장기적으로 snapshot graph와 retention policy가 필요하다.

## 0.1.0 문서 대비 업데이트해야 할 결론

0.1.0 문서의 결론은 "Firecracker 기반 volatile execution plane의 핵심 경로가 가능하다"에 가까웠다.

0.2.0 문서의 결론은 다음처럼 바뀐다.

- ephemera는 이제 단일 host에서 VM 생성, task 실행, 종료, snapshot, restore, profile 분리, agent proxy까지 갖춘 실행 평면이다.
- PoC 단계를 넘어 single-host feature complete로 볼 수 있다.
- 다만 multi-tenant production system으로 보기에는 quota, egress control, audit, snapshot lifecycle policy, concurrent restore semantics가 아직 부족하다.

## 다음 단계 제안

0.3.0에서 우선순위를 둘 만한 영역은 다음과 같다.

- snapshot retention과 garbage collection
- same-snapshot concurrent restore semantics 확정
- egress policy와 네트워크 격리
- tenant별 quota와 VM lifecycle policy
- audit log와 task execution trace
- agent token rotation과 snapshot secret scrubbing
- OpenAPI 스펙 또는 MCP adapter 계약 문서화
